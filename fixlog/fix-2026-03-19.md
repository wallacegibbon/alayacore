# Bug Fix: Text Messages Lost on Save

**Date**: 2026-03-19

## Issue

When saving a session, all Text messages (TU, TA tags) were lost while Function messages (FC, FR) were preserved. This only happened once, not consistently.

## Root Cause

In `internal/llm/agent.go`, when tool calls were present, the code created a new assistant message containing ONLY the tool calls:

```go
assistantMsg := Message{
    Role:    RoleAssistant,
    Content: toolCallsToContent(toolCalls), // Only tool calls, text lost!
}
```

The `stepMessages` from `StepCompleteEvent` (which contains BOTH text AND tool calls from the provider) was being ignored when tool calls were present.

## Fix

Use the complete assistant message(s) from `stepMessages`:

```go
if len(stepMessages) == 0 {
    stepMessages = []Message{{
        Role:    RoleAssistant,
        Content: toolCallsToContent(toolCalls),
    }}
}
allMessages = append(allMessages, stepMessages...)
```

This handles 0, 1, or N messages properly and is consistent with the text-only code path.

## Why It Was Intermittent

The bug only occurred when:
1. The LLM returned BOTH text content AND tool calls in the same response
2. The session was saved after such a response

If the LLM returned only text or only tool calls, the bug didn't manifest.

## Data Flow Explanation

### Provider Accumulates ALL Content

When the LLM streams its response, the provider (e.g., OpenAI) accumulates everything in a state object:

```go
type openAIStreamState struct {
    textBuilder      strings.Builder     // Accumulates text: "Let me help"
    reasoningBuilder strings.Builder     // Accumulates reasoning
    toolCalls        []llm.ToolCallPart  // Accumulates tool calls
}
```

### Provider Builds Complete Message

When the stream ends, the provider calls `getMessage()` which packages everything into one message:

```go
func (s *openAIStreamState) getMessage() llm.Message {
    content := make([]llm.ContentPart, 0, 2+len(s.toolCalls))
    
    // Add reasoning (if any)
    if s.reasoningBuilder.Len() > 0 {
        content = append(content, llm.ReasoningPart{...})
    }
    
    // Add text (if any)
    if s.textBuilder.Len() > 0 {
        content = append(content, llm.TextPart{
            Type: "text",
            Text: s.textBuilder.String(),  // "Let me help with that."
        })
    }
    
    // Add tool calls (if any)
    for _, tc := range s.toolCalls {
        content = append(content, tc)
    }
    
    return llm.Message{
        Role:    llm.RoleAssistant,
        Content: content,  // ALL content: text + reasoning + tool calls
    }
}
```

### Provider Sends StepCompleteEvent

The provider sends this complete message via `StepCompleteEvent`:

```go
eventChan <- llm.StepCompleteEvent{
    Messages: []llm.Message{state.getMessage()},  // Complete message!
    Usage:    state.getUsage(),
}
```

So `StepCompleteEvent.Messages[0]` contains:
```
Message{
    Role: "assistant",
    Content: [
        TextPart{"Let me help with that."},
        ToolCallPart{ToolName: "get_weather", ...}
    ]
}
```

### Agent Receives in processStreamEvents

The agent's `processStreamEvents()` receives this and stores it:

```go
case StepCompleteEvent:
    stepMessages = e.Messages  // Complete message with text AND tool calls
    stepUsage = e.Usage
```

### The Bug Was Here

**BEFORE the fix**, the code ignored `stepMessages` when tool calls were present:

```go
// BUG: Ignoring stepMessages, creating NEW message with ONLY tool calls
assistantMsg := Message{
    Role:    RoleAssistant,
    Content: toolCallsToContent(toolCalls),  // Only tool calls, text lost!
}
```

**AFTER the fix**, we use the complete message from the provider:

```go
// FIX: Use the complete message(s) from provider (has text AND tool calls)
if len(stepMessages) == 0 {
    stepMessages = []Message{{
        Role:    RoleAssistant,
        Content: toolCallsToContent(toolCalls),
    }}
}
allMessages = append(allMessages, stepMessages...)
```
